# EchoMind — 补充分析报告

> **项目背景**：EchoMind 是一款本地优先的"灵感备忘录"，定位"快速记 + 偶尔深挖"。架构上明确对标 Obsidian Sync 模式，分三层（详见 [`docs/architecture-hybrid-cloud.md`](docs/architecture-hybrid-cloud.md)）：
> - **L1 Core**（必装、永久免费）：桌面 Tauri + React + 本地 SQLite + sqlite-vec 向量检索 + 多 LLM 提供商接入
> - **L2 Bridge**（可选订阅、约 ¥20-30/月）：用户选定"上云子集"经 VPS 与微信 ClawBot（腾讯官方 iLink 协议，npm `@tencent-weixin/openclaw-weixin`）互通，桌面离线时手机仍可全功能查询/录入
> - **L3 Sync**（未实现，未来订阅）：多桌面端 E2E 加密同步
>
> 当前 v0.1 已落地 L1 完整 + L2 的 Phase 1–3（VPS 服务端、本地对接、`/chat` LLM 转发），**Phase 4 部署与稳定性进行中**（Docker Compose + nginx TLS、`/bridge/thoughts/capture` 路由、`/chat` 速率限制、断线重连）。
>
> **MVP 范围**：
> - **做**：单人公开发布版本，主推 L1 + L2 订阅制；桌面 Win/Mac 双端；微信 bot 自动同步；AI enrich + 总结 + 对话；本地语义检索；用户自备 LLM Key（BYO Key），可选推送至 VPS 供 `/chat` 远程执行（带预算硬上限默认 $1/天）
> - **不做**：协作功能；企业版；移动端原生 App；自训练模型；安卓 / iOS 原生分发；E2E Sync（L3，留到下一阶段）；Persona 胶囊 / 代用户回复朋友（已明确作废）
> 分析日期：2026-05-12

## 一、核心假设与验证框架

> 以下假设按"若不成立则项目方向需调整"的标准筛选。当前项目已 v0.1 实现，验证视角是"能否走向公开分发与订阅变现"，而非"技术能否跑起来"。

### H1：用户对"现有笔记工具的记录摩擦"痛点强到愿意切换 ❓

| 字段 | 内容 |
|------|------|
| **假设内容** | 目标用户（重度笔记 / 多端记录场景）认为现有方案（Notion / Obsidian / Flomo / 微信收藏）的记录摩擦足够大，愿意为一款新工具尝试切换或并行使用 |
| **为什么重要** | 这是整个项目的需求侧前提。若用户实际把 Flomo 当作"够好"，EchoMind 没有市场机会 |
| **已有证据** | 作者本人的真实痛点（Flomo 经常忘开、微信收藏路径长）；竞品（立知、Pile.app 等）持续涌现说明市场存在；公开渠道未做用户访谈 |
| **验证动作** | (1) 4–6 周内访谈 10–15 名重度笔记用户，问"现在记灵感最痛的环节"开放题；(2) 在小红书 / X / 即刻发种草帖，看自然评论里是否有共鸣；(3) 灰度发布给 30 名种子用户跟踪 30 天留存 |
| **判定标准** | (1) 访谈中 ≥ 60% 主动提到"路径长 / 切设备 / 忘了"；(2) 种子用户 30 天留存 ≥ 40%（行业基准 25–30%） |
| **若不成立** | 重新定位场景（如转向"读书摘录" / "工作 OKR 速记"等更具体的细分场景），或放弃公开分发回归自用 |
| **当前判断** | ⭐⭐ 仅作者样本，未做外部验证 |
| **验证时机** | POC 第 1–2 周（访谈），第 4–8 周（种子用户留存） |

### H2："本地优先 + 微信原生 + 用户控制上云子集"三角组合是差异化核心 ❓

| 字段 | 内容 |
|------|------|
| **假设内容** | "本地优先（数据所有权） + 微信 ClawBot（腾讯官方 iLink 协议）原生集成 + 用户自选上云子集（时间窗 / 标签 / 来源组合）"这个三角组合在中文市场没有直接竞品，且这种"明确隐私代价"的 Obsidian Sync 风格能赢得理性用户 |
| **为什么重要** | 决定 EchoMind 能否在拥挤的笔记赛道里讲出独特故事。任一边塌陷都失去差异化：去掉本地优先就退化为 Notion 仿品；去掉微信就退化为 Obsidian 仿品；去掉子集控制就和 Flomo 同质 |
| **已有证据** | Obsidian 无官方微信集成（社区插件复杂且不稳），但其本地 + 付费 Sync 模式经市场验证；Notion 微信入口需依赖第三方机器人且体验割裂；Flomo 有微信集成但无桌面端、无 AI、无本地数据控制；架构文档 §10 明确以 Obsidian Sync 为参照系 |
| **验证动作** | (1) 写一篇对比测评（EchoMind vs Obsidian + 微信插件 vs Notion + 第三方 bot vs Flomo），实测四方"地铁上记一条→晚上桌面整理"流程的耗时与摩擦点；(2) 访谈中专门问 "你愿意为'明确告诉你哪些数据上云、哪些不上'的微信集成付费吗"；(3) 在隐私敏感型用户群（独立开发者社区、技术博主）做小范围灰度 |
| **判定标准** | (1) 实测 EchoMind 端到端耗时比次优方案 ≥ 30% 优；(2) 访谈中 ≥ 50% 用户表示"双端无缝 + 子集控制"是切换关键因素；(3) 隐私敏感细分用户群中"愿意付费"比例 ≥ 25%（高于普通用户） |
| **若不成立** | 收缩到纯桌面 + 强本地 AI 方向（与 Obsidian 拉开本地 AI 差距），放弃微信桥这一重资产；或反向收缩为"重微信轻桌面"对标 Flomo |
| **当前判断** | ⭐⭐⭐ L1 + L2 技术全部实现且决策日志清晰，市场反馈未验证 |
| **验证时机** | POC 第 2–3 周（对比测评），持续跟踪 |

### H3：BYO Key 模式 + VPS 基础设施成本能让订阅 ¥20-30 闭环 ⚠️

| 字段 | 内容 |
|------|------|
| **假设内容** | 架构上采用 BYO Key（用户自带 LLM Key）模式后，订阅费仅需覆盖 VPS 基础设施 + 少量带宽 + 维护时间分摊；¥20-30/月订阅价（架构文档 §8 建议值）足以让 100 付费用户量级闭环 |
| **为什么重要** | 决定订阅模式的财务可行性。若架构层面把 LLM 成本转嫁用户，平台侧成本结构大幅简化；但用户接受度需验证（"我已经付了订阅费，为什么还要自己买 Key"是合理质疑） |
| **已有证据** | 架构文档 §8 明确 VPS ¥15-25/月 + 建议订阅 ¥20-30/月；架构 §5.4 明确 LLM key 推送至 VPS 是可选项（默认不推），用户始终对 Key 有控制力；预算硬上限 $1/天已实现（`add_usage` 原子停用） |
| **验证动作** | (1) 用 200 条真实灵感跑完整 enrich + 周总结流程，按各家 API 计价表算实际成本（基线参考，让用户有数）；(2) 给 10 名种子用户跑 30 天，统计 BYO Key 模式下实际人均 LLM 成本分布；(3) 跑两组对照：A 组讲"订阅 ¥30，含 LLM"，B 组讲"订阅 ¥20 + 自备 Key"，看转化率与心理接受度差异 |
| **判定标准** | (1) BYO Key 用户月 LLM 实测成本中位数 ≤ ¥8（让"自备 Key"在心理上比订阅含费划算）；(2) BYO Key 模式下试用→订阅转化率 ≥ 4%（不显著低于含费模式）；(3) VPS 单实例承载 200 用户时延迟与稳定性达标 |
| **若不成立** | (1) 引入分层订阅（基础 ¥20 BYO / Pro ¥50 含费）兼顾两类心智；(2) 推出"启动包"含 ¥10 LLM 额度降低首次使用门槛；(3) 启用本地小模型做 enrich（`fastembed` 已集成，可避开外部 API 成本） |
| **当前判断** | ⭐⭐⭐⭐ 架构决策已锁定 BYO Key + 预算上限，定价合理性需市场验证 |
| **验证时机** | POC 第 0 周（VPS 单用户保本线核算），第 2–3 周（成本模拟），第 4–8 周（种子用户实测 + AB 对照） |

| 字段 | 内容 |
|------|------|
| **模型使用方式** | 直接调用 API（zero-shot / few-shot）。Enrich 用结构化 JSON output；总结用 long-context one-shot；`/chat` 走多轮对话。选择原因：开发成本低、零数据需求、可随时切换 provider；BYO Key 让 EchoMind 平台不锁定特定模型，用户可按需切换 OpenAI / Claude / Gemini / DeepSeek |
| **所需数据量估算** | 无训练数据需求。Eval 集需要 50–100 条人工评估的"理想 enrich 输出"作为质量基线；针对 BYO Key 心智需要 30–50 名用户的真实成本反馈 |
| **效果验证标准** | (1) Enrich 输出在 domain / tags 字段上与人工标注一致率 ≥ 75%；(2) 周总结的"主线提炼"在盲评中得分 ≥ 4/5；(3) 用户主观成本满意度（NPS 类问题）≥ 7/10 |

### H4：sqlite-vec 在 5K~10K 灵感量级下检索质量与延迟达标 ⚠️

| 字段 | 内容 |
|------|------|
| **假设内容** | 在单用户 5K–10K 条灵感量级下，sqlite-vec 的语义检索 P95 延迟 ≤ 200ms，召回 top-5 中相关项命中率 ≥ 80% |
| **为什么重要** | 关联发现 / 语义检索是产品体验的隐性底层。若延迟卡顿或召回不准，"第二大脑"价值崩塌 |
| **已有证据** | sqlite-vec 官方 benchmark 在 100K 向量量级仍维持 < 100ms；EchoMind 当前实测库小（< 500 条），尚未压力测试 |
| **验证动作** | (1) 合成 / 导入 10K 条真实文本数据生成 embedding，跑 1000 次随机查询测延迟分布；(2) 人工标注 50 个查询的相关项基线，对比 EchoMind 召回结果 |
| **判定标准** | (1) P95 延迟 ≤ 200ms，P99 ≤ 500ms；(2) Top-5 召回命中率 ≥ 80% |
| **若不成立** | 引入分层检索（关键词预过滤 + 向量精排）；切换到 Qdrant / Chroma 等专用向量库；调小 embedding 维度（512 → 384） |
| **当前判断** | ⭐⭐⭐⭐ 技术上有强信心，但未做规模验证 |
| **验证时机** | POC 第 2–3 周 |

| 字段 | 内容 |
|------|------|
| **数据获取渠道** | 评估数据可由作者历史笔记 + 公开数据集（如知乎问答语料、微博情感数据集片段）拼凑得到；门槛低 |
| **数据质量风险** | 合成数据的语义分布可能与真实灵感笔记差异大，导致 benchmark 数字偏乐观 |
| **备选方案** | 待种子用户阶段用真实数据回归一次 benchmark |

### H5：AI 自动 enrich 输出质量足以让用户信任不修改 ❓

| 字段 | 内容 |
|------|------|
| **假设内容** | LLM 自动生成的 context / domain / tags 在 ≥ 70% 场景下用户接受不修改，不会引发"AI 帮倒忙"的反感 |
| **为什么重要** | EchoMind "记下就走，AI 帮你整理"是核心叙事。若用户每次都要手动修正，叙事崩塌且记录摩擦不降反升 |
| **已有证据** | 作者自用体验中 enrich 大多可接受；少数边角案例（如纯英文短句、emoji 多的内容）表现差 |
| **验证动作** | (1) 取 100 条多样化灵感（短 / 长 / 中英混 / 纯链接 / 截图说明等），用人工逐条评估 enrich 质量；(2) 在种子用户阶段记录每条灵感的"enrich 修改率" |
| **判定标准** | (1) 人工评估接受率 ≥ 70%；(2) 种子用户实际修改率 ≤ 30% |
| **若不成立** | (1) 引入 enrich 前的"快速预览 + 一键放弃"按钮，让用户主动选择是否 enrich；(2) Prompt 调优（增加 few-shot 示例）；(3) 给用户配置"风格偏好"以引导 enrich 输出 |
| **当前判断** | ⭐⭐⭐ 主观感觉不错，缺乏定量证据 |
| **验证时机** | POC 第 2–3 周（人工评估），第 4–8 周（种子用户回归） |

| 字段 | 内容 |
|------|------|
| **模型使用方式** | 直接调用 API + 强约束 prompt（结构化输出）。考虑过 fine-tuning 但放弃：数据量不够，且通用模型已达可用线 |
| **所需数据量估算** | 无训练数据需求。Eval 集 100 条手工标注 |
| **效果验证标准** | (1) Tags 与人工标注 Jaccard 相似度 ≥ 0.6；(2) Domain 一致率 ≥ 75%；(3) Context 摘要在盲评中得分 ≥ 3.5/5 |

### H6：单人维护下，Tauri + Rust + Node.js 三套运行时复杂度可持续 ❓

| 字段 | 内容 |
|------|------|
| **假设内容** | 在单人开发节奏下（约 5–10 小时 / 周），EchoMind 三套运行时的维护成本（依赖升级、bug 修复、新功能迭代）能维持月度小迭代节奏不退化 |
| **为什么重要** | 决定项目能否长期存活。技术栈过重会让维护成本指数级增长，3–6 个月后开发节奏会停滞，用户流失 |
| **已有证据** | 当前 v0.1 在约 3 个月内完成；Tauri / Rust 学习曲线已度过；Node.js bot 使用腾讯官方 npm 包 `@tencent-weixin/openclaw-weixin`，断更风险低 |
| **验证动作** | (1) 跟踪未来 3 个月每周开发时间 vs 产出功能数 vs 已知 bug 数的趋势；(2) 跑一次"重大依赖升级演练"（如 Tauri 2.x → 2.y）记录耗时 |
| **判定标准** | (1) 月度发布节奏不退化（每月至少 1 个有意义的小迭代）；(2) Tauri 大版本升级耗时 < 2 周 |
| **若不成立** | (1) 砍掉 VPS bridge 模块（最重的部分），改用第三方 BaaS；(2) Bot 转为可选插件由社区维护；(3) 引入 1 个兼职协作者 |
| **当前判断** | ⭐⭐⭐ 当前节奏健康，但 3 个月样本太短 |
| **验证时机** | 持续跟踪（每月底回顾） |

### H7：L2 Bridge 订阅 ¥20-30 / 月的付费意愿能覆盖单人维护成本 ❓

| 字段 | 内容 |
|------|------|
| **假设内容** | 在公开发布后，L1 Core 永久免费的前提下，目标人群中至少 5% 愿意为 L2 Bridge（微信桥 + 上云子集 + 离线手机查询）订阅付费（¥20-30 / 月，架构 §8 建议价），订阅收入能覆盖 VPS 摊销 + 维护时间机会成本 |
| **为什么重要** | 决定项目能否商业化养活自己。L1 免费用户基数 vs L2 付费转化率的乘积决定营收。BYO Key 模式（H3）已转嫁 LLM 成本，订阅压力主要来自 VPS 与维护工时 |
| **已有证据** | Obsidian Sync ¥10/月、Publish ¥20/月模式跑通（直接对标）；Flomo ¥99/年模式跑通；架构文档 §8 直接建议 ¥20-30；EchoMind 自身无任何付费数据 |
| **验证动作** | (1) 公开发布前做 landing page + 付费意向调研（"如果 L2 Bridge 订阅 ¥25/月你会订阅吗"），收集 100+ 反馈；(2) 早期用户提供 14 天免费试用 L2，看转化率（架构含"停止订阅 = VPS 立即销毁"承诺，可作为增信）；(3) 算清单用户保本线（VPS 分摊 ÷ 付费用户数 + 维护工时单价 ÷ 用户数） |
| **判定标准** | (1) 调研中"愿意付费 L2"比例 ≥ 15%；(2) 试用→订阅转化率 ≥ 5%；(3) 保本线 ≤ ¥10 / 用户 / 月（订阅价 ¥20-30 时毛利率 ≥ 50%） |
| **若不成立** | (1) 探索一次性买断（¥99–199）替代订阅；(2) 把 L3 Sync（多桌面端 E2E 同步）提前到下一阶段，做"L2+L3 套餐"提升订阅性价比；(3) 收缩为纯本地工具放弃 Cloud Bridge，回归 L1 单层免费模型 |
| **当前判断** | ⭐⭐ 价格段有竞品锚定，但付费转化无外部信号 |
| **验证时机** | POC 第 0 周（保本线核算），第 1–2 周（landing page + 调研），第 4–8 周（试用转化） |

### H8：微信 ClawBot（iLink 协议）作为 2026 腾讯官方放开的个人号 Bot API，长期可用性可控 ✅

| 字段 | 内容 |
|------|------|
| **假设内容** | EchoMind 接入的是 2026 年腾讯官方主动放开的个人号 Bot 协议——产品名「微信 ClawBot」、底层协议「iLink（智联）」、入口 `ilinkai.weixin.qq.com`、npm 包 `@tencent-weixin/openclaw-weixin`（`@tencent-weixin` 是腾讯官方 npm scope），是 OpenClaw AI Gateway 的官方 Channel Plugin。无商用风险、无封号风控。剩余次要风险仅在限流策略 / 协议大版本升级 / 长期 SLA 上，且 EchoMind 客户端已对常见约束（session 过期 / 长轮询超时 / 消息去重 / 游标）做处理 |
| **为什么重要** | 此项澄清后，把 H8 从原"灰色协议合规风险"框架降级为"官方 SDK 用法风险"，整体优先级大幅下降。微信桥接（H2 差异化核心载体）的法律 / 合规基础因此变得很硬 |
| **已有证据** | (1) `@tencent-weixin/openclaw-weixin` npm 包发布在腾讯官方 scope 下，2026 年活跃维护中；(2) 文档明确"WeChat officially opened a legal personal Bot API through OpenClaw"；(3) bot 账号独立于用户主号，加 bot 是加新 contact，不影响主号；(4) EchoMind 当前实现已稳定运行多月，bot_token 可重复登录 |
| **验证动作** | (1) 阅读 OpenClaw 官方文档与 npm 包 README，记录限流 / 配额上限；(2) 在 v0.1 公开发布前用 staging 跑 50 用户并发，确认在限流以内；(3) 跟踪 npm 包大版本升级（当前 2.0.x 对应 OpenClaw ≥ 2026.3.22），订阅 GitHub release notify |
| **判定标准** | (1) 文档化的限流上限允许单用户 ≥ 100 条消息/天；(2) 近 6 个月内无大版本 breaking change 风险；(3) staging 50 用户并发无被限流 |
| **若不成立**（小概率） | (1) 出现严苛限流时，切到 OpenClaw 多账号扩展（如 ImGoodBai/WeClawBot-ex）支持多 bot 横向扩容；(2) 大版本 breaking change 时按 OpenClaw 的 dist-tag（latest / legacy）跟踪兼容；(3) 极端情况下回退到企业微信备选 |
| **当前判断** | ⭐⭐⭐⭐⭐ 官方协议 + 官方维护 + 官方 npm 包，几乎不构成战略风险 |
| **验证时机** | POC 第 0 周（阅读官方限流文档），第 4–6 周（staging 压测顺带验证） |

### H9：L2 Bridge Phase 4 部署与稳定性收尾能在公开发布前完成 ⚠️

| 字段 | 内容 |
|------|------|
| **假设内容** | 架构文档 §7 列出的 Phase 4 未完成项（Docker Compose + nginx TLS 部署配置 / `/bridge/thoughts/capture` bridge 模式新建想法 / `/chat` 速率限制 / Budget 超限 Tauri 事件通知 / bridge 客户端断线重连指数退避）能在公开发布前 4–6 周内全部收尾，且不引发新一轮架构返工 |
| **为什么重要** | 这些不是优化项，是面向多用户公开服务的"门槛项"。缺 TLS 部署 = 不能上生产；缺 capture 路由 = bridge 模式无法发文字录入；缺速率限制 = 一个用户的失控调用可耗尽自己预算；缺断线重连 = 桌面网络抖动引发数据不一致 |
| **已有证据** | 架构文档 §7 Phase 1–3 已 ✅ 标注完成；Phase 4 五项未完，但都是常规工程问题、无新技术风险；当前 v0.1 已能跑通 happy path |

| **判定标准** | (1) Phase 4 五项全部交付且 staging 跑 14 天无 P0 故障；(2) 50 并发下 `/bridge/*` API P95 ≤ 500ms；(3) 主动断线 100 次后 bridge 客户端自动恢复率 100% |
| **若不成立** | (1) 收缩公开发布范围到"L1 + 桌面 only"，L2 Bridge 标注 beta；(2) 引入 1 名兼职帮把 Phase 4 收尾；(3) 将公开发布日期顺延 4 周专门做稳定性 |
| **当前判断** | ⭐⭐⭐ 范围明确、技术可控，瓶颈是单人开发节奏（链接 H6） |
| **验证时机** | POC 第 0 周（拆 issue + 估排期），第 4–6 周（staging 跑测） |

## 三、竞品技术路线

| 维度 | EchoMind | Obsidian | Notion |
|------|----------|----------|--------|
| **存储** | 本地 SQLite + 可选 VPS 同步 | 本地 Markdown 文件 + 第三方同步 | 云端独占 |
| **数据所有权** | 用户完全掌控 | 用户掌控本地文件 | 受 Notion 控制 |
| **AI 能力** | 内置（自动 enrich / 总结 / 对话） | 需第三方插件 | Notion AI（独立订阅） |
| **检索** | 本地向量 + 关键词 | 关键词 + 部分插件支持向量 | 云端关键词 + AI Search |
| **微信集成** | 一等公民（官方桥） | 无（社区插件不稳） | 无（依赖第三方 bot） |
| **多端速记** | 桌面热键 + 微信 + 托盘 | 桌面 + 移动端 App，无微信 | Web + App，无微信 |
| **协作** | 不做 | 受限 | 强 |
| **订阅价** | 待定（计划 ¥30–50/月） | 永久免费 / Sync ¥10/月 / Publish ¥20/月 | ¥10–80/月（多档） |
| **目标场景** | 个人灵感速记 + 偶尔深挖 | 知识管理 + 个人 Wiki | 团队协作 + 工作台 |
| **差异化窗口** | 桌面 + 微信 + 本地优先 + 内置 AI 的组合在中文市场无直接竞品 | — | — |

**关键结论**：EchoMind 的差异化窗口聚焦在「中文用户 + 微信 ClawBot 官方原生集成 + 本地优先 + 用户控制上云子集 + 内置 AI」五个维度同时成立的交集，单独任一维度都有更强对手。

**模式参照系**：架构上明确以 Obsidian Sync 为参照——核心永久免费、Sync/Bridge 是付费可选、明确告诉用户隐私代价（不假装零知识）、退订即销毁云端数据。这套模式在英文社区已被市场验证，EchoMind 的赌注是它在中文笔记市场同样成立。

## 四、产品形态

**核心载体：跨平台桌面 App + 微信 Bot + 可选 VPS 服务**

- **桌面端**（Windows / macOS）：Tauri 2.0 + React，主操作面板，托盘常驻 + 全局热键速记
- **微信 Bot**（Node.js daemon）：手机端无负担入口，命令式交互 + 自然语言新建
- **VPS Cloud Bridge**（Rust + axum，可选）：桌面离线时由 VPS 接管微信 Bot；上云子集存储；LLM 请求转发

**未来形态延伸（不在 MVP）：**
- 移动端 PWA（仅查看 + 简易速记）
- 浏览器扩展（网页摘录直接进灵感库）
- 本地小模型 Embedding / Enrich（隐私模式）

## 五、补充影响因素

### 人才
- **当前**：单人开发（前端 + Rust + Node.js + 产品 + 运营全栈），约 5–10 小时/周
- **风险**：单点故障；任何技术栈深坑（如 Tauri 大版本升级）都吞掉数周
- **缓解**：与 H6 联动，设置月度回顾点；预留 Bot 模块解耦给社区维护的可能性

### 客户导入
- **目标人群**：重度笔记用户、独立产品爱好者、有微信 + 桌面双场景的知识工作者
- **冷启动渠道**：小红书 / X / 即刻 / 少数派 / 独立产品社区（IndieHackers 中文圈）
- **种子用户**：作者社交圈 + 早期内测群（30 人量级）
- **风险**：付费工具在中文市场冷启动天然慢；需配套至少 1 篇深度对比测评作为弹药

### 法律合规
- **微信 ClawBot（iLink 协议）**：2026 年腾讯官方放开的个人号 Bot API，npm 包 `@tencent-weixin/openclaw-weixin` 在腾讯官方 scope 下维护；无第三方协议路线的封号 / 灰色合规问题，无商用风险；剩余仅是限流配额这类常规 SDK 用法风险（见 H8）
- **用户数据存储**：本地优先架构降低主要合规风险；L2 Bridge 上云子集需写明 (a) 存储位置（VPS 部署地）(b) 加密方式（架构 §5.5 注明 `sync_key` 推送加密、VPS 落盘前解密以支持搜索——这是明确的隐私代价）(c) 销毁机制（停止订阅 = 数据立即销毁，已实现 `/bridge/terminate`）。架构 §4 已有"知情同意"提示文案模板，公开发布前需法律审查
- **LLM 提供商 ToS**：Gemini / DeepSeek / OpenAI / Claude 各家关于"代用户调用"的条款不同。BYO Key + 用户主动授权推送 Key 至 VPS 的设计已大幅降低平台代理风险，但 `/chat` 路径仍需逐家确认转发模式合规性
- **跨境数据**：若 VPS 部署在境外，涉及个人信息出境合规（PIPL）；若部署在境内，备案 + ICP 是前置条件
- **付费协议**：订阅条款需明确"L1 永久免费 / L2 月费可随时退 / 退订立即销毁云端数据" 三条核心承诺，与架构决策保持一致
